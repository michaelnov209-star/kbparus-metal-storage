### Сильные стороны

| Разделение ответственности | Отлично | Данные в `data/`, логика в `lib/`, UI в `components/` и `app/` |

### Архитектурные решения Codex (сохранены)

- **Нет базы данных:** весь контент — TypeScript-файлы. Упрощает деплой, усложняет редактирование контента без разработчика
- **Калькулятор на клиенте:** вся логика в браузере, быстро для UX, нет сервер-зависимости
- **6 профилей калькулятора** соответствуют реальным типам оборудования КБ Парус
- **Пилот-категория:** только первая категория (auto-sheet-metal) имеет продуктовые страницы — правильное MVP-решение

## 3. Технический аудит

### 3.1 Критические проблемы (влияют на SEO или доступность)

#### P1 — Нет robots.txt и sitemap.xml
**Статус:** ✅ ИСПРАВЛЕНО в этой сессии  
**Было:** Поисковики не имели инструкций по индексации, нет карты сайта  
**Стало:** `public/robots.txt` + динамический `app/sitemap.ts` (22+ URL)  

#### P1 — `animation-timeline: view()` не поддерживается в Safari
**Статус:** ✅ ИСПРАВЛЕНО в этой сессии  
**Было:** Элементы с классом `.reveal` могли оставаться невидимыми в Safari (нет поддержки Scroll-Driven Animations API)  
**Стало:** `@supports` wrapper + базовый fallback `opacity: 1; transform: none`  
**Затронуто:** все секции главной страницы с классом `.reveal`

---

### 3.2 Высокие проблемы (влияют на надёжность)

#### P2 — Нет `.env.example`
**Статус:** ✅ ИСПРАВЛЕНО в этой сессии  
**Проблема:** Новый разработчик не знает, какие переменные окружения нужны  
**Стало:** `.env.example` с `BITRIX24_WEBHOOK_URL` и комментарием  

#### P2 — Нет `vercel.json`
**Статус:** ✅ ИСПРАВЛЕНО в этой сессии  
**Проблема:** Статические ассеты (изображения, видео) кешируются по дефолту Vercel, без явного контроля  
**Стало:** `vercel.json` с `Cache-Control: public, max-age=31536000, immutable` для `/assets/`  

#### P2 — `LeadForm` в секции контактов отправляет пустой `calculatorInput`
**Статус:** Задокументировано, не исправлено (требует архитектурного решения)  
**Проблема:** `components/LeadForm.tsx` отправляет `calculatorInput: {}`, но пользователь мог до этого заполнить калькулятор. Данные калькулятора теряются при отправке из раздела "Контакты"  
**Рекомендация:** Передавать state калькулятора через context или URL params. Реализация в отдельном спринте.

### 3.3 Средние проблемы (влияют на поддерживаемость)

#### P3 — `ffmpeg-static` в devDependencies
**Статус:** ✅ ИСПРАВЛЕНО в этой сессии  
**Проблема:** Пакет нигде не импортируется в коде (использовался для обрезки видео вручную). Загрязняет зависимости, увеличивает `npm install` на CI  

#### P3 — Мёртвые Pexels URL в `globals.css`
**Статус:** ✅ ИСПРАВЛЕНО в этой сессии  
**Проблема:** Классы `.hero` и `.final-cta` содержали `url("https://images.pexels.com/...")`, но оба класса не используются в `page.tsx` (заменены на `line-hero` и другие)  
**Риск:** Браузер не загружал эти URL (CSS не применялся), но они создавали потенциальную зависимость от внешнего сервиса  

#### P3 — `HeroVisual.tsx` импортируется, но не используется
**Статус:** Задокументировано  
**Проблема:** Компонент существует и импортируется в `page.tsx` (`import { HeroVisual }`), но в JSX не рендерится — видео встроено напрямую в `page.tsx`  
**Риск:** Путаница для новых разработчиков. Не вызывает ошибок, TypeScript не жалуется (неиспользуемые импорты не запрещены)  
**Рекомендация:** Удалить компонент и импорт в следующем рефакторе  

#### P3 — `recommendStorageSystem` не используется в цепочке расчётов
**Статус:** Задокументировано  
**Описание:** `lib/calculator/recommendation.ts` экспортирует `recommendStorageSystem()`, но `lib/calculator/pricing.ts` строит рекомендации самостоятельно внутри `calculateStorageSystem()`. Функция `recommendStorageSystem` вызывается в `Calculator.tsx` отдельно — это не баг, но архитектура неочевидна для нового разработчика  

---

### 3.4 Низкие проблемы (технический долг)

#### P4 — `weight-scale` опция = 3 ₽
**Статус:** Задокументировано (отмечено в README)  
**Описание:** В `data/storageSystems/excelCalculator.ts` опция "Весы на распалетчик" имеет price = 3 — очевидно placeholder. Нужна реальная цена от отдела продаж КБ Парус  

#### P4 — Цена показывается только "от X"
**Описание:** В калькуляторе нет диапазона цен. Это архитектурное решение (соответствует `PROJECT_CONTEXT.md`: "не обещать точную цену"), но технически можно добавить верхнюю границу через factor × 1.15-1.25  

#### P4 — Нет error boundaries
**Описание:** Если `Calculator.tsx` или `CatalogGrid.tsx` упадут с runtime ошибкой, пользователь увидит пустую страницу. React Error Boundary остановит каскад. Требует UX-решения (что показывать при ошибке)  

#### P4 — Нет loading skeleton
**Описание:** При клиентской навигации страница пуста до гидратации. Для производительности Next.js App Router это минимально, но `loading.tsx` файлы добавят полировку

## 4. Аудит калькулятора

### Архитектура
```
excelCalculator.ts (профили) → validation.ts (нормализация) → pricing.ts (расчёт) → Calculator.tsx (UI)
                                                            ↘ recommendation.ts (UX рекомендации, отдельно)
```

### 6 профилей калькулятора

| ID | Название | Тип логики |
|----|----------|-----------|
| `auto-sheet-metal` | Автоматические стеллажи для листа | `automatic` |
| `auto-sort-metal` | Автоматические стеллажи для сорта/трубы | `automatic` |
| `rollout-cassette-rack` | Выкатные кассетные стеллажи (одностороннее) | `rollout` |
| `forklift-cassette-rack` | Кассетные стеллажи под вилочный погрузчик | `forkliftCassette` |
| `two-side-rollout-rack` | Двусторонние выкатные стеллажи | `rollout` |
| `hybrid-rollout-rack` | Гибридные стеллажи (вилочный + выкатной) | `hybrid` |

### Формулы по типам

**automatic:**  
`fromPrice = towers × (towerBase + shelves × shelfBase × loadFactor × dimensionFactor)`  
`+ consoles × consoleBase`  
`+ (mounting ? 180000 : 0)`  
`+ (delivery ? 95000 + towers × 15000 : 0)`  
`+ (painting ? 120000 × towers : 0)`  

**forkliftCassette:**  
`fromPrice = towers × towerBase + shelfCount × shelfPrice`  

**rollout:**  
`fromPrice = towers × towerBase + shelfCount × shelfPrice + gateCount × 80000`  

**hybrid:**  
`fromPrice = towers × (forkliftBase + rolloutShelves × rolloutPrice + forkliftShelves × forkliftPrice)`  

### Тест-кейсы (все проходят)

| # | Тип | Ожидаемая цена |
|---|-----|----------------|
| 1 | auto-sheet-metal | 7 318 000 ₽ |
| 2 | auto-sort-metal | 21 387 840 ₽ |
| 3 | rollout-cassette-rack | 1 783 819 ₽ |
| 4 | forklift-cassette-rack | 567 450 ₽ |
| 5 | two-side-rollout-rack | 890 155 ₽ |
| 6 | hybrid-rollout-rack | 1 516 679 ₽ |
| 7 | Округление formatRoundedRub | 34 429 000 ₽ |

### Найденные аномалии калькулятора

1. **weight-scale = 3 ₽** — placeholder, нужна реальная стоимость
2. **`calculatorInput: {}` в LeadForm** — state из калькулятора не передаётся в форму контактов
3. **`recommendStorageSystem` экспортируется, но вызывается отдельно** — не баг, но неочевидная архитектура

## 5. Аудит CRM-интеграции

`app/api/leads/route.ts`:
- POST `/api/leads` принимает `{ contact, city, comment, calculatorInput, utm }`
- Нормализует и пересчитывает конфигурацию через `calculateStorageSystem`
- Формирует payload для Bitrix24 `crm.deal.add`
- При наличии `BITRIX24_WEBHOOK_URL` — отправляет в CRM
- Без переменной — mock-режим, возвращает `{ ok: true, mode: "mock" }`

**Что не реализовано:**
- UTM-параметры (поле передаётся, но не заполняется из frontend)
- Дедупликация лидов
- Telegram уведомления
- Email уведомления менеджеру
