# Handoff для Claude Code: kbparus-metal-storage

Этот файл нужен, чтобы Claude Code мог продолжить работу без длинного контекста переписки.

## Проект

- Сайт: «Системы хранения металла» для КБ Парус.
- Stack: Next.js App Router, React, TypeScript, lucide-react.
- Production: https://kbparus-metal-storage.vercel.app
- GitHub: https://github.com/michaelnov209-star/kbparus-metal-storage
- Vercel project: `kbparus-metal-storage`

## Главная бизнес-логика

Сайт должен быть не просто каталогом, а инструментом первичного подбора:

1. Клиент выбирает тип оборудования.
2. Выбирает ходовые параметры из утвержденных значений.
3. Видит стартовую стоимость `от ... ₽`.
4. Оставляет заявку.
5. Менеджер/инженер получает структурированные данные для расчета.

Важно: сайт не обещает точную цену. Формулировка в интерфейсе: стартовая стоимость, ориентир, финальная стоимость уточняется после инженерной проверки.

## Excel-источник

Актуальный файл: `C:/Users/micha/Downloads/Калькулятор-New.xls`

Правила:

- Лист `Главная` = публичная структура каталога.
- Лист `Админка` = модель будущей админки, где меняются цены, коэффициенты и опции.
- Остальные формульные листы = отдельные профили калькулятора.
- `1` и `0` в Excel означают имитацию выбора пользователя. В React это должно быть нормальное состояние, а не копирование 0/1 в UI.
- Не выводить на сайт слова `Excel`, ячейки, названия листов и сокращения руководителя.
- Не брать старые данные про печи, полимеризацию или линии порошковой окраски.

## Калькулятор

Публичный UI:

- 3 шага: выбор оборудования, параметры, результат.
- Параметры выбираются кнопками/карточками, не свободными полями.
- Нет слов `Excel`, `sourceSheet`, адресов ячеек, `SKU`, `product_id`, `external_1c_id`.
- Итог: `от ... ₽`.

Внутренняя логика:

- Профили лежат в `data/storageSystems/excelCalculator.ts`.
- Формулы лежат в `lib/calculator/pricing.ts`.
- API пересчитывает данные на сервере в `app/api/leads/route.ts`.

## Визуальные правила

- Основной акцент: `#fc5413`.
- Стиль: современный промышленный B2B, не дешевый лендинг.
- Hero должен быть сильным: темный фон, крупный заголовок, метрики, CTA.
- Фото должны реально отображаться, а не быть alt-подписями.
- Можно использовать временные remote-фото, но они должны быть легко заменяемыми в `visualAssets.ts`.

- Не показывать публичный блок конкурентов.
