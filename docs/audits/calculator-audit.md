## 1. Архитектура калькулятора

```
Пользователь вводит параметры
         ↓
components/Calculator.tsx (UI, 3 шага)
         ↓
lib/calculator/validation.ts → normalizeCalculatorInput()
         ↓
lib/calculator/pricing.ts → calculateStorageSystem()
         ↓
data/storageSystems/priceFactors.ts (коэффициенты)
data/storageSystems/excelCalculator.ts (профили)
         ↓
lib/calculator/recommendation.ts → recommendStorageSystem() (UX рекомендации, отдельно)
         ↓
Результат: fromPrice, lineItems, engineeringSummary, recommendedConfig
         ↓
app/api/leads/route.ts (при отправке заявки — пересчёт на сервере)
```

| Параметр | Значения по умолчанию |
|----------|----------------------|
| Высота | 3000 мм |
| Ширина | 1500 мм |
| Длина | 3100 мм |
| Нагрузка | 1500 кг |
| Башен | 2 |
| Полок/башню | 10 |

**Тип логики:** `automatic`  
**Формула:**
```
fromPrice = towers × (towerBase × dimensionFactor
          + shelfCount × shelfBase × loadFactor)
          + consoleCount × consoleBase
          + options (painting, mounting, delivery)
```

**Базовые цены (из `priceFactors.ts`):**
- Автоматическая полка: 75 000 ₽

## 5. Найденные проблемы и рекомендации

### Проблема 1: `weight-scale` = 3 ₽

**Где:** `data/storageSystems/excelCalculator.ts`, профиль `auto-sheet-metal`, опция `weight-scale`  
**Описание:** Цена опции "Весы на распалетчик" = 3 ₽. Это очевидный placeholder.  
**Влияние:** При выборе этой опции цена увеличивается на 3 ₽ — незаметно для пользователя, но некорректно.  
**Решение:** Получить реальную стоимость от отдела продаж КБ Парус и обновить значение в файле.

---

### Проблема 2: `LeadForm` отправляет пустой `calculatorInput`

**Где:** `components/LeadForm.tsx`  
**Описание:** Компонент `LeadForm` (используется в разделе "Контакты" главной страницы) отправляет `calculatorInput: {}` — пустой объект. Пользователь мог до этого заполнить весь калькулятор, но эти данные не попадают в заявку из раздела контактов.  
**Влияние:** Менеджер КБ Парус получает заявку без информации о выбранном оборудовании.  
**Решение:**
- Вариант A (простой): Сохранять `calculatorInput` в `localStorage` при каждом изменении в калькуляторе, читать в `LeadForm`
- Вариант B (архитектурный): Поднять state калькулятора в `app/page.tsx`, передать в `LeadForm` через props

---

### Проблема 3: `recommendStorageSystem` — двойная логика

**Где:** `lib/calculator/recommendation.ts` + `lib/calculator/pricing.ts`  
**Описание:** Функция `recommendStorageSystem()` определяет тип системы на основе введённых параметров. Но `calculateStorageSystem()` в `pricing.ts` строит рекомендацию самостоятельно без вызова `recommendStorageSystem()`. В `Calculator.tsx` оба вызываются отдельно.  
**Влияние:** Не баг. Обе функции дают одинаковый результат для пользователя. Но архитектура неочевидна.  
**Решение (долгосрочно):** Вызывать `recommendStorageSystem()` внутри `calculateStorageSystem()` и включать результат в `CalculatorResult`.

---

### Наблюдение: Нет диапазона цен "от–до"

**Где:** `lib/calculator/pricing.ts` → `Calculator.tsx`  
**Описание:** Калькулятор показывает `"от X ₽"` без верхней границы. Технически это корректно (соответствует `PROJECT_CONTEXT.md`: "не обещать точную цену").  
**Вариант улучшения:** Показывать диапазон "от X ₽ до Y ₽" где Y = X × 1.15–1.25. Это улучшит конверсию — клиент понимает ценовой диапазон.
