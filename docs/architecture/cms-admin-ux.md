# CMS Admin UX — архитектура и план развития

Документ объединяет: (1) текущее состояние Payload-админки, (2) исторический выбор CMS и сравнение альтернатив, (3) UX-предложения и план развития.

---

## 1. Текущее состояние (то, что уже работает)

### Цель админки

Payload admin работает как **операционный центр управления сайтом**, а не как набор технических таблиц. Маркетолог и менеджеры могут самостоятельно править описания, цены, добавлять товары и фото без участия разработчика.

### Логика навигации

Коллекции и глобалы сгруппированы по бизнес-разделам в `payload/admin/structure.ts`:

| Раздел | Содержит | Назначение |
|--------|----------|------------|
| Главная страница | `home-content`, `site-navigation` | Модульное управление лендингом: hero, преимущества, кейсы, отзывы, FAQ, партнёры, баннеры |
| Каталог | `categories`, `subcategories`, `products` | 17 категорий, подкатегории, товары; URL-поля помечены как чувствительные для SEO |
| Калькулятор | `calculator-profiles` | Профили, цены, коэффициенты и опции, влияющие на стоимость и Telegram-заявку |
| Компания | `contacts` | Контакты, реквизиты, мессенджеры, сквозные данные |
| Медиа | `media` | Фото, видео, баннеры, логотипы, документы |
| Заявки | `lead-management` | Операционная карта форм, Telegram и подготовка Bitrix24 |
| Пользователи | `users` | Сотрудники админки, роли и доступы |

### Что сделано для менеджеров

- Dashboard «Центр управления сайтом и каталогом» с быстрыми переходами (статический server component, без клиентского JS).
- Labels и descriptions переведены на деловой русский язык.
- Главная страница описана как редактор лендинга по порядку блоков.
- Медиа-библиотека получила поля «Тип материала», «Область использования», «Заметки редактора».
- Пользователи: раздельные поля «имя», «фамилия», «отображаемое имя», «должность».
- Раздел «Заявки: формы и интеграции» — без секретов и без изменения env.

### Роли пользователей

В `payload/collections/Users.ts`:

| Роль | Что может |
|------|-----------|
| `admin` | Всё, включая управление пользователями |
| `editor` | Контент, медиа, без управления пользователями и интеграциями |
| `photographer` | Только Media-коллекция |

## 2. Performance-замеры и оптимизации

### Профиль до оптимизации

После простоя production admin requests занимали:

- `/admin` — около 6.6 с
- `/admin/collections/products` — около 6.5 с
- `/api/health` — около 7.0 с
- `/` (главная) — около 1.0 с

### Диагностика

Профиль указывает не на CSS или bundle как главную причину, а на **serverless cold start и подключение к Neon/Payload runtime**. Optimизация CSS/JS не даст ощутимого эффекта без warm-keepalive.

### Применённые безопасные оптимизации

- Убраны тяжёлые relationship columns из списков товаров и подкатегорий.
- Ограничены `defaultLimit` в list views.
- Добавлены `listSearchableFields` для предсказуемого поиска.
- Dashboard сделан статическим server component.
- Hero-видео заменено на оптимизированный MP4 без изменения URL.

### Следующие шаги по перформансу

- Uptime ping `/api/health` каждые 5 минут (keep warm).
- Если задержки остаются критичными — платный Neon tier (Compute Auto-suspend off).
- После стабилизации UX — переход к полноценной CMS-моделизации страниц, форм и Bitrix24.

## 3. Выбор CMS — почему Payload (исторический контекст)

В мае 2026 года рассматривались четыре варианта CMS. Финальный выбор — **Payload CMS 3**.

### Сравнение альтернатив

| Решение | Плюсы | Минусы | Решение |
|---------|-------|--------|---------|
| **Sanity Studio** | Бесплатно до 3 редакторов, отличный UX, live preview, real-time | US-SaaS, оплата в USD из РФ нестабильна, контент на их серверах | **Отклонено** — санкционный риск |
| **Decap CMS** (Netlify CMS) | Open-source, контент в git, нет внешних зависимостей | Слабый UX, нет live preview, каждое изменение = коммит = деплой ~2 мин, картинки в репо | **Отклонено** — маркетолог не сможет редактировать 30+ строк таблиц коэффициентов через GitHub PR |
| **Strapi / Directus** | Большое сообщество, готовые плагины | Требуют отдельный сервер, тяжёлый docker-контейнер, не TypeScript-first | Рабочие, но громоздкие |
| **Payload CMS 3** | TypeScript-first, живёт внутри Next.js, контроль над данными, бесплатно | Самостоятельная поддержка, нужна Postgres | **Выбран** |

### Почему Payload победил

1. **Российская юрисдикция данных.** Postgres в Neon (через Vercel Marketplace) + Vercel Blob — всё под управлением компании.
2. **TypeScript-first** — идеально для нашего стека Next.js + TS strict.
3. **Excel-калькулятор с 32 листами и таблицами коэффициентов** — Payload позволяет редактировать каждую таблицу через формы, не через GitHub PR.
4. **Vercel-совместимость** — Payload 3 встраивается в App Router как часть Next.js-приложения, не требует отдельного хостинга.
5. **Стоимость первого года: 0 ₽** (Vercel Hobby + Neon Free + Vercel Blob Free).
6. **Перенос на купленный домен работает без проблем** — Postgres и Blob независимы от домена. Если когда-нибудь захочется уйти с Vercel: `pg_dump` + копия Blob = переезд за полдня на любой VPS.

- Hero: заголовок, подзаголовок, метрики, видео.
- «Что можно хранить» — карточки материалов.
- «До/после склада» — тексты и фото.
- Преимущества — карточки.
- Кейсы — фото, текст, метрики.
- География поставок — точки на карте + список городов.
- Отзывы.
- «О компании» — текст + метрики.
- Баннеры на kbparus.ru и линииокраски.рф.

- FAQ.
- Контакты (телефоны, email, адрес, соцсети, рабочие часы).

### Глобально
- SEO-метаданные каждой страницы.
- Текст согласия на обработку персональных данных.
- Хедер, футер, легальные ссылки (`site-navigation`).

### Что НЕ выносится в CMS (остаётся в коде)
- Формулы калькулятора в `lib/calculator/pricing.ts` — физика и инженерия, не маркетинговый текст.
- Discriminated `pricing.kind` профиля — меняет ветку формулы, маркетолог видит read-only.
- API заявок и интеграция с Telegram/Bitrix24.
- Защита форм (honeypot, rate-limit).

## 5. Сценарии пользователей

### Маркетолог: смена описания категории

1. Логин в `/admin`.
2. Раздел «Каталог» → «Категории» → выбор категории.
3. Правка описания и SEO-полей.
4. «Сохранить» → через 60 секунд (ISR `revalidate=60`) изменения видны на сайте.

### Фотограф: загрузка партии новых фото

1. Логин (роль `photographer`).
2. Раздел «Медиа» → «Загрузить».
3. Указать тип материала и область использования.
4. Назначить фото нужным товарам через коллекцию `products`.

### Менеджер: добавление нового товара

1. Раздел «Каталог» → «Товары» → «Создать».
2. Заполнить 7 tabs (main / images / price / configurator / specs / SEO / publishing).
3. Опционально привязать `calculator-profile`.
4. Опубликовать (поле `_status: published`).

### Главный редактор: смена баннера на главной

1. Раздел «Главная страница» → `home-content` → секция «Баннеры».
2. Заменить изображение, обновить ссылку.
3. Сохранить → автообновление страницы через ISR.

## 6. Roadmap внедрения

### Этап 1 — Базовая стабилизация (выполнено)
- Bootstrap Payload, коллекции и глобалы.
- Build pipeline с preflight checks, schema push, importMap regen.
- Production deploy, `/api/health` ok, `cms.ok: true`.

### Этап 2 — Seed контента (выполнено)
- 17 категорий и подкатегории через `cms:seed-catalog`.
- `home-content` с 15 секциями через `cms:seed-home-content`.
- 47 медиа-ассетов через `cms:seed-media`.
- Навигация и футер через `cms:seed-site-navigation`.

### Этап 3 — UX-полировка админки (выполнено / в работе)
- Группировка коллекций.
- Русские labels и descriptions.
- Dashboard «Центр управления».
- Список безопасных performance-оптимизаций.

### Этап 4 — Calculator hardening (планируется)
- Payload field validators: `value > 0`, factor 0.1..10, монотонный порядок размеров.
- Кнопка «Тестовый расчёт» в админке — показывает цену по сохранённым данным до публикации.
- Daily CI: Vitest против production-данных через Local API → Telegram alert, если pinned тесты поплыли.

### Этап 5 — Lead operations (планируется)
- Полная CMS-моделизация форм и интеграций.
- Управление статусами заявок прямо в админке.
- Подготовка к Bitrix24 cutover.

### Этап 6 — Обучение и документация
- 30-минутный screencast на русском: редактирование товара, замена фото, FAQ, что НЕ трогать в калькуляторе.
- 1-страничный cheat-sheet.
- 2 практических edits с пользователем + 1 соло.

## 7. Принципы UX-решений в админке

1. **Бизнес-язык, не технический.** Labels пишутся как «Заголовок hero-секции», не «hero.title».
2. **Группировка по сценариям, не по моделям данных.** «Главная страница» — это всё, что собирается в один лендинг, а не отдельные коллекции/глобалы.
3. **Sensible defaults.** Новые поля опциональны или имеют разумные дефолты; сайт продолжает работать.
4. **Видимость следующего действия.** Каждый сценарий редактирования имеет понятный «следующий шаг» — сохранить, опубликовать, проверить на сайте.
5. **Не трогать формулы из админки.** Бизнес-логика (формулы калькулятора, доставка лидов) остаётся в коде. CMS управляет только тем, что меняется руками маркетолога.
