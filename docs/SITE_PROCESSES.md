# Процессы сайта «Системы хранения металла»

## 1. Контент и каталог

Источник текущей структуры каталога — Excel-файл `Калькулятор-New.xls`, лист `Главная`.

Процесс обновления:

1. Ответственный меняет структуру ассортимента в утвержденном источнике.
2. Разработчик переносит публичные разделы в `data/storageSystems/excelCatalog.ts`.
3. На сайте не показываются адреса ячеек, названия листов и внутренние комментарии.
4. После изменения проверить, что каталог содержит только утвержденные разделы.

Будущая CMS должна редактировать те же поля: `id`, `title`, `summary`, `scenario`, `image`, `featured`.

Каждый раздел каталога должен иметь рабочую страницу `/catalog/[id]`. Карточки на главной, hero-ссылки и выпадающее меню "Каталог" ведут на эти страницы, чтобы пользователь мог углубиться в категорию и оставить заявку по конкретному направлению.

## 2. Калькулятор стоимости

Публичный процесс:

1. Клиент выбирает тип оборудования.
2. Выбирает ходовые размеры, нагрузку, количество полок, башен и опции.
3. Видит сумму `от ... ₽`.
4. Заполняет контакты.
5. Заявка уходит в `/api/leads`.

Технический процесс:

1. UI в `components/Calculator.tsx` хранит состояние выбора.
2. Данные профилей берутся из `data/storageSystems/excelCalculator.ts`.
3. `lib/calculator/pricing.ts` рассчитывает стоимость и инженерные параметры.
4. API повторно нормализует и пересчитывает входные данные на сервере.

Правило: расчеты не добавлять в React-компоненты. Все формулы должны оставаться в `lib/calculator`.

## 3. Заявки и CRM

Endpoint: `app/api/leads/route.ts`

Принимает:

- `contact`
- `city`
- `comment`
- `calculatorInput`
- `recommendedConfig`
- `preliminaryPriceFrom`
- `utm`

Режимы:

- Без `BITRIX24_WEBHOOK_URL`: mock-response.
- С `BITRIX24_WEBHOOK_URL`: отправка в Bitrix24.

Будущие действия:

- Согласовать пользовательские поля Bitrix24.
- Добавить UTM parsing.
- Добавить события аналитики для шагов калькулятора.

## 4. Фото и визуальные ассеты

Текущие изображения лежат как remote URL в `data/storageSystems/visualAssets.ts`.

Процесс замены:

1. Добавить реальные фото/видео КБ Парус в `public/assets/images/`.
2. Обновить пути в `visualAssets.ts`.
3. Проверить production, что изображения реально видны.

Для изображений разделов каталога использовать `public/assets/images/catalog/` и подключать путь в `data/storageSystems/excelCatalog.ts`. Текущие пользовательские файлы:

- `01-auto-sheet-metal.jpg`
- `03-sort-and-pipe-storage.jpg`
- `04-manual-sort-and-pipe-storage.webp`

Ключевые блоки с изображениями:

- hero
- каталог
- страницы категорий каталога
- что можно хранить
- до/после склада
- кейсы
- калькулятор summary
- баннеры

## 5. География поставок

Сейчас точки заданы в `app/page.tsx` как массив `geoProjects` с процентными координатами.

Будущий CMS-процесс:

- хранить город;
- название компании или тип проекта;
- координаты на карте или город для геокодирования;
- короткое описание проекта.

## 6. FAQ и клиентские тексты

FAQ должен говорить языком клиента:

- какая система подойдет;
- как хранить трубы/лист/длинномер;
- как считается нагрузка;
- можно ли под размеры помещения;
- доставка, монтаж, расширение;
- передача заявок в CRM/1С.

Не выводить в FAQ технические поля вроде `SKU`, `product_id`, `external_1c_id`.

## 7. Деплой

Перед деплоем:

```bash
npm run lint
npm run test
npm run build
```

Git:

```bash
git status
git add <только нужные файлы>
git commit -m "..."
git push origin main
```

Vercel:

```bash
npx vercel@latest deploy --prod --yes --scope michaelnov209-3230s-projects
```

После деплоя:

- открыть production URL;
- проверить калькулятор;
- проверить `/api/leads`;
- убедиться, что не создан новый Vercel project.

## 8. Передача между Codex и Claude Code

Правило очередности:

1. Агент завершает итерацию.
2. Обновляет документацию.
3. Делает commit и push.
4. Следующий агент делает `git pull origin main`.
5. Следующий агент читает `README.md`, `CHANGELOG.md`, `docs/CLAUDE_CODE_HANDOFF.md` и этот файл.

Так оба агента работают с одним состоянием проекта и не затирают изменения друг друга.
