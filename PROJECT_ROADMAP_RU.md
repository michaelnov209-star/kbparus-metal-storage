## Спринт 1 — Конверсия и CRM (ПРИОРИТЕТ: ВЫСОКИЙ)

**Цель:** Превратить сайт из витрины в реальный инструмент лидогенерации.

### 1.1 Bitrix24 webhook — КРИТИЧНО

**Что:** Настроить `BITRIX24_WEBHOOK_URL` в Vercel Environment Variables.  
**Зачем:** Сейчас все заявки теряются (API работает в mock-режиме, данные не попадают в CRM).  
**Как:** Получить webhook URL от КБ Парус → добавить в Vercel Dashboard → Environment Variables.  
**Кто:** Менеджер или разработчик + доступ к Bitrix24 аккаунту КБ Парус.  
**Сложность:** Низкая (10 минут).  
**Риск:** Нулевой для кода.

### 1.2 UTM-трекинг в форме заявки

**Что:** Читать UTM-параметры из URL и передавать в `POST /api/leads`.  
**Зачем:** Отдел маркетинга не знает, с каких каналов идут заявки.  
**Файлы:** `components/LeadForm.tsx` + `components/Calculator.tsx`  
**Как:** `new URLSearchParams(window.location.search)`, передать в payload.  
**Сложность:** Низкая (2-3 часа разработки).

### 1.3 Calculator state → LeadForm

**Что:** Передавать заполненные данные из калькулятора в форму контактов (раздел "Контакты" внизу страницы).  
**Зачем:** Сейчас `LeadForm` в разделе "Контакты" отправляет `calculatorInput: {}` — данные о выбранном оборудовании теряются.  
**Файлы:** `app/page.tsx`, `components/LeadForm.tsx`  
**Как:** Поднять state калькулятора в `page.tsx` или использовать `localStorage` как fallback.  
**Сложность:** Средняя (4-6 часов).

### 1.4 Реальные фотографии КБ Парус

**Что:** Заменить стоковые Pexels-фото в `data/storageSystems/visualAssets.ts` на реальные фото оборудования и объектов.  
**Зачем:** B2B-клиенты доверяют реальным фото производства, не стоку.  
**Файлы:** `data/storageSystems/visualAssets.ts`, папка `public/assets/images/`  
**Кто:** Фотограф или маркетолог КБ Парус.  
**Сложность:** Зависит от наличия материалов.

### 1.5 Telegram-уведомления на заявки

**Что:** При получении заявки отправлять сообщение в Telegram-бот менеджера.  
**Зачем:** Быстрая реакция без постоянного мониторинга Bitrix24.  
**Файлы:** `app/api/leads/route.ts`  
**Как:** `fetch` к Telegram Bot API в том же route handler.  
**Сложность:** Низкая (2-3 часа).

## Спринт 2 — Контент и каталог (ПРИОРИТЕТ: СРЕДНИЙ)

**Цель:** Расширить каталог, добавить продуктовые страницы для всех категорий.

### 2.1 Продуктовые страницы для остальных 16 категорий

**Что:** Сейчас только первая категория (auto-sheet-metal) имеет продуктовые страницы.  
**Зачем:** 16 категорий ведут на заглушку "страница в разработке".  
**Файлы:** `data/storageSystems/catalogDepth.ts`, `app/catalog/[id]/[productId]/page.tsx`  
**Как:** Добавить данные о продуктах для каждой категории.  
**Сложность:** Высокая (требует контента — описаний, характеристик, изображений от КБ Парус).

### 2.2 Admin UI / CMS для редактирования контента

**Что:** Простой интерфейс для редактирования карточек продуктов без разработчика.  
**Зачем:** Сейчас все изменения требуют деплоя. Маркетолог не может обновить цену или описание.  
**Варианты:**
- **Contentful/Sanity** — headless CMS, интеграция с Next.js (рекомендуется)
- **Tina CMS** — git-based, подходит для небольших проектов
- **Кастомный Admin** — максимальный контроль, высокая стоимость разработки  
**Сложность:** Высокая.

### 2.3 Исправить weight-scale цену

**Что:** Опция "Весы на распалетчик" имеет price = 3 ₽ (placeholder).  
**Файл:** `data/storageSystems/excelCalculator.ts`  
**Кто:** Отдел продаж КБ Парус предоставляет реальную стоимость.  
**Сложность:** Минимальная (одна строка кода после получения цены).

### 2.4 Диапазон цен "от X до Y" в калькуляторе

**Что:** Показывать верхнюю границу цены (например, `fromPrice × 1.20` как максимум).  
**Зачем:** "от 1 500 000 ₽ до 1 800 000 ₽" лучше конвертирует, чем просто "от 1 500 000 ₽".  
**Файлы:** `lib/calculator/pricing.ts`, `lib/calculator/types.ts`, `components/Calculator.tsx`  
**Сложность:** Средняя (изменение типов + UI).

## Спринт 3 — Аналитика и SEO (ПРИОРИТЕТ: СРЕДНИЙ)

**Цель:** Понимать, что делают пользователи и оптимизировать под поисковики.

### 3.1 Google Analytics 4 / Yandex Metrika

**Что:** Добавить счётчик(и) в `app/layout.tsx`.  
**Зачем:** Сейчас нет данных о посетителях, времени на странице, источниках трафика.  
**Файлы:** `app/layout.tsx`  
**Как:** `next/script` с `strategy="afterInteractive"`.  
**Сложность:** Низкая (1-2 часа).

### 3.2 Отслеживание событий калькулятора

**Что:** Событие при каждом переходе по шагам калькулятора, выборе профиля, отправке заявки.  
**Зачем:** Понять, на каком шаге калькулятора клиенты уходят.  
**Файлы:** `components/Calculator.tsx`  
**Как:** `gtag('event', ...)` или `ym(id, 'reachGoal', ...)`.  
**Сложность:** Низкая (3-4 часа после установки GA4).

### 3.3 SEO-контент для категорий

**Что:** Уникальные тексты (1000+ слов) для каждой из 17 страниц категорий.  
**Зачем:** Сейчас `description` в `generateMetadata()` — короткие (до 160 символов). Без контента страницы не ранжируются по длинным запросам.  
**Файлы:** `data/storageSystems/excelCatalog.ts` или отдельный `content.ts` per category  
**Кто:** Копирайтер + SEO-специалист + данные от КБ Парус.  
**Сложность:** Высокая (контент, а не код).

### 3.4 Структурированные данные (Schema.org)

**Что:** JSON-LD разметка: `Organization`, `Product`, `FAQPage`.  
**Зачем:** Rich snippets в поиске Google (звёздочки, FAQ в выдаче).  
**Файлы:** `app/layout.tsx`, `app/catalog/[id]/page.tsx`, `app/page.tsx`  
**Сложность:** Средняя (4-6 часов).

## Спринт 4 — UX-полировка (ПРИОРИТЕТ: НИЗКИЙ)

**Цель:** Довести UX до уровня premium industrial brand.

### 4.1 Мобильное бургер-меню

**Что:** На мобильных устройствах nav скрывается за иконкой-гамбургером.  
**Зачем:** Текущий header на мобильных занимает ~30% экрана.  
**Файлы:** `app/page.tsx`, `app/globals.css`  
**Сложность:** Средняя (4-6 часов, включая анимацию).

### 4.2 Error boundaries

**Что:** React Error Boundary вокруг `Calculator`, `CatalogGrid`, форм.  
**Зачем:** При crash показывать "Что-то пошло не так, обновите страницу" вместо пустого экрана.  
**Сложность:** Средняя (требует решения по UI сообщений).

### 4.3 Loading skeletons

**Что:** Skeleton placeholder пока компоненты гидратируются.  
**Зачем:** Убрать "мигание" при первой загрузке страницы на медленных соединениях.  
**Файлы:** `app/` — добавить `loading.tsx` файлы  
**Сложность:** Низкая (2-3 часа).

### 4.4 Микроанимации на hover-состояниях

**Что:** Плавные переходы при наведении на карточки каталога, кнопки CTA.  
**Зачем:** Ощущение premium-интерактивности.  
**Файлы:** `app/globals.css`  
**Сложность:** Низкая (1-2 часа).

## Долгосрочно (6+ месяцев)

### 1С / WMS интеграция

**Что:** Синхронизация каталога и наличия оборудования из 1С напрямую на сайт.  
**Зачем:** Актуальные данные о наличии и ценах без ручного обновления.  
**Сложность:** Очень высокая. Требует API 1С, middleware, webhook-синхронизацию.

### Личный кабинет для B2B клиентов

**Что:** Авторизация, история заявок, статус проекта, документы.  
**Зачем:** Repeat-покупатели (промышленные предприятия обычно приобретают несколько систем).  
**Сложность:** Очень высокая. Требует auth (NextAuth), база данных (PostgreSQL), роли.

### 3D-конфигуратор

**Что:** Интерактивная 3D-модель стеллажа на основе введённых параметров.  
**Зачем:** Дифференциация от конкурентов, "wow-фактор" для технических специалистов.  
**Технологии:** Three.js или Babylon.js, GLTF-модели.  
**Сложность:** Очень высокая.
